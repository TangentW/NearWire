# Implementation Round 1 Remediation

Date: 2026-07-12

This record maps every actionable Round 1 implementation finding to the final implementation and evidence. Historical review reports remain unchanged; this file records the subsequent resolution.

| Finding | Resolution | Verification |
| --- | --- | --- |
| Architecture 1: a claim in progress could survive generation cancellation or pause/resume | The manager now installs the reservation and attempt under its lock before channel construction, tracks active generation IDs, and revalidates the exact attempt after claim. Listener ingress is deactivated synchronously, and a stale claim cannot reinsert itself. | `testClaimInProgressCannotSurviveGenerationCancellationOrPauseResume` holds claim behind a deterministic barrier and covers both cancellation and pause/resume. |
| Architecture 2 and Correctness 4: shutdown/reset had no bounded cleanup receipt | Admission stop now returns one idempotent receipt that awaits exact connection-core cancellation and handed-off cleanup. Window close, retry, reset, and AppKit termination wait up to one second without reopening admission; reset begins Keychain work only afterward. | `testCleanupReceiptCompletesOrTimesOutWithoutReopeningAdmission`, `testIdentityResetWaitsForAdmissionCleanupReceipt`, and updated lifecycle termination coverage pass. |
| Architecture 3: synchronous local-network failure lost its typed recovery | Synchronous listener construction preserves only `SecureTransportError.localNetworkUnavailable`; every other error remains collapsed to the fixed listener category. | `testSynchronousLocalNetworkListenerFailureKeepsRecoverableCategory` and the separate asynchronous failure test pass. |
| Architecture 4: focused secure-transport evidence overstated skipped tests | The implementation evidence now reports only the result of the exact command that was run. Fresh final validation is recorded below and under `implementation-validation.md`. | Evidence wording no longer attributes skipped tests to a passing command. |
| Correctness 1 and Security 3: reset could delete a referenced foreign certificate or accept a missing key | Destructive reset validates the complete owned tuple before certificate deletion: metadata version, persistent reference, certificate digest and serial, exact permanent nonexportable private key, fixed profile, self-signature/trust, public-key digest, and key correspondence. Explicit partial reset fails closed. Automatic repair never deletes a certificate whose ownership is not proven. | Injected lifecycle tests cover missing/wrong keys, adversarial foreign certificates, digest mismatches, certificate/key/metadata delete failures, and surviving items. Real foreign-certificate and malformed-metadata tests also pass. |
| Correctness 2 and Security 2: the planned data-protection Keychain path had no executable zero-configuration evidence | The reviewed product contract now intentionally uses the standard per-user macOS login Keychain with `kSecUseDataProtectionKeychain=false`, synchronization disabled, no sharing entitlement, and non-interactive reads/deletes. Apple's data-protection Keychain access groups require provisioning-profile-authorized signing entitlements, which conflicts with the internal Viewer's ad-hoc/local zero-configuration requirement. The injectable persistence boundary proves the complete state machine, and the real login-Keychain integration exercises the production selector mode. | Fresh Viewer XCTest result: 44 passed, 0 failed, 0 skipped. The real lifecycle creates and reloads a persistent identity, proves nonexportability and `SecIdentity` assembly, and performs both resets. See Apple's [TN3137](https://developer.apple.com/documentation/Technotes/tn3137-on-mac-keychains) and [`kSecUseDataProtectionKeychain`](https://developer.apple.com/documentation/security/ksecusedataprotectionkeychain). |
| Correctness 3: deadline and terminal races depended on wall time | Admission now injects a monotonic scheduler. Deadline tests advance a manual scheduler; accept-versus-timeout exercises both winner orders. Listener races use explicit gates where ordering matters. | `testAcceptAndTimeoutChooseExactlyOneTerminalWinnerInBothOrders`, deterministic silent/partial/approval deadline tests, and claim-in-progress tests pass. |
| Correctness 5: certificates stopped working once a 3,650-day lifetime crossed 2050 | DER validity uses canonical UTCTime through 2049 and GeneralizedTime from 2050 onward; parsing is strict for both forms. | `testCertificateBuilderSupportsUTCAndGeneralizedTimeValidityWindows` covers the transition and post-2040 issuance through Security validation. |
| Security 1: incoming connections could allocate an unbounded MainActor task backlog before the 32-slot gate | Incoming listener callbacks now enter one synchronous lock-protected generation gate before any MainActor hop. Stale, paused, stopped, and over-capacity wrappers are rejected there. Pending UI snapshots are latest-only and schedule at most one MainActor drain task per burst. | `testListenerAdmissionIngressBoundsBurstBeforeMainActorWork` proves 32 admitted wrappers, immediate rejection of excess work, and bounded UI delivery. |
