# Pre-Implementation Security, Performance, and Documentation Review

## Scope

Independently reviewed the complete `viewer-application-foundation` proposal, design, capability specifications, tasks, existing Core secure listener/channel and trust boundaries, repository architecture and security documentation, and current Apple primary guidance. The review focused on self-signed identity lifecycle, Keychain and reset safety, the narrow X.509 plan, Bonjour disclosure, TLS wording, pre-session resource bounds, privacy and entitlements, and proportionality. No proposal, design, specification, task, production, or test artifact was modified; this report is the only added file.

## Findings

### High — The 32-attempt and 15-second bounds do not cover pre-Hello admission

The design limits confirmation UI to 32 valid attempts for 15 seconds, but only after one App `WireHello` has been decoded (`design.md:59-65`; `specs/viewer-application-foundation/spec.md:82-88`). Default automatic admission has no stated concurrent-attempt cap, and neither policy gives the same total deadline to connections stalled before Hello. The existing Core listener emits an incoming wrapper for every accepted `NWConnection` while its listener gate is open; it has no process-wide concurrent incoming limit (`SecureByteChannel.swift:747-760`). Per-channel frame and send limits do not bound the number of channels, Tasks, sockets, or pre-Hello receives.

A nearby faulty or hostile client can therefore open many connections and stall TLS/channel/Hello progress without ever occupying one of the 32 confirmation rows. Refresh briefly overlaps listener generations, so a per-listener limit would also permit the total to multiply. This contradicts the proposal's claim of bounded pre-session admission and creates a local resource-exhaustion path before the future session manager exists.

**Required remediation:** define one Viewer-runtime-wide budget covering every accepted but not terminally handed-off connection, from incoming-wrapper claim through TLS readiness, bounded Hello decode, optional confirmation, and consumer handoff. Claim a slot before constructing a channel or Task; reject immediately when all 32 slots are owned; start one 15-second monotonic total deadline at claim for both automatic and confirmation policies; and release the exact slot once on handoff, rejection, timeout, pause, replacement, failure, or shutdown. The budget must be shared by overlapping old/replacement listeners. Add deterministic tests for stalled pre-Hello attempts, automatic policy, refresh overlap, full-cap rejection without channel creation, timeout, and shutdown returning connection/Task/slot counts to zero.

### Medium — The X.509, Keychain, and reset lifecycle is not exact enough to implement safely

The artifacts choose a reasonable narrow P-256/SHA-256 self-signed leaf, but do not define its validity period, renewal behavior, serial-number construction, exact subject/issuer contents, allowed extensions, or load-time trust/expiry validation (`design.md:39-45`). The existing App trust evaluator applies Basic X.509 policy and `SecTrustEvaluateWithError`, so certificate signature and validity must pass. A certificate that merely parses and matches its private key can still be expired or not yet valid. The promise that TLS identity remains stable until reset is therefore incomplete without an expiry/renewal rule.

“Non-exported Keychain private key” and “versioned service/tag names” also need exact Security attributes and class-scoped queries. Apple documents that a macOS data-protection Keychain operation should set `kSecUseDataProtectionKeychain=true`, and that a macOS private key with extractability disabled cannot be externally represented ([`kSecUseDataProtectionKeychain`](https://developer.apple.com/documentation/security/ksecusedataprotectionkeychain), [`SecKeyCopyExternalRepresentation`](https://developer.apple.com/documentation/security/seckeycopyexternalrepresentation(_:_:))). Without exact class, application tag/label, access group, synchronization, accessibility, and deletion selectors, partial repair or reset can miss an owned item or delete a similarly named foreign item.

The rollback statement is also inaccurate: it says Viewer Keychain items can be removed by application uninstall (`design.md:89-91`), while Keychain deletion on uninstall is not a contractual macOS lifecycle. The only specified UI reset deletes TLS material and intentionally preserves the installation identifier, so it cannot remove all Viewer identity data.

**Required remediation:** before implementation, define one fixed non-identifying X.509 profile: version, positive random serial bounds, exact P-256 SPKI and ECDSA-with-SHA256 identifiers, fixed subject/issuer fields containing no pairing/installation value, validity window and clock-skew policy, CA=false, digital-signature/server-auth use, and no network-fetching extensions. Define load-time signature, current-validity, key-correspondence, and `SecIdentity` checks plus one bounded repair/renewal rule. Set the persistent private key nonextractable and sensitive, use the data-protection Keychain without synchronization or a shared access group, and document exact selectors for installation, private-key, and certificate items. Reset must close listener/admission first, delete only those exact TLS selectors, and fail closed on partial deletion/recreation. Add foreign-item preservation, export-failure, expired/not-yet-valid, partial delete, and reset interruption tests. Correct the uninstall claim and provide a distinct, confirmed way or operator procedure to remove the installation identifier when all Viewer identity data must be erased.

This does not require a general ASN.1 framework or a new runtime dependency. The private bounded DER encoder remains proportionate if its complete profile is fixed and real Security parsing, trust evaluation at boundary dates, key matching, and a live TLS handshake validate its output.

### Medium — Viewer privacy, local-network declarations, and sandbox entitlements are undecided

Task 2.1 mentions app metadata and entitlements, but the normative artifacts do not state the exact `Info.plist`, entitlements, signing assumption, or Viewer-owned privacy manifest. Apple says any app using local networking or Bonjour should provide `NSLocalNetworkUsageDescription`, and apps registering or browsing specific Bonjour services should list them in `NSBonjourServices` ([`NSLocalNetworkUsageDescription`](https://developer.apple.com/documentation/bundleresources/information-property-list/nslocalnetworkusagedescription), [`NSBonjourServices`](https://developer.apple.com/documentation/bundleresources/information-property-list/nsbonjourservices), [TN3179](https://developer.apple.com/documentation/technotes/tn3179-understanding-local-network-privacy)). A sandboxed macOS listener requires `com.apple.security.network.server`; TCP accepted under the server entitlement remains bidirectional ([server entitlement](https://developer.apple.com/documentation/bundleresources/entitlements/com.apple.security.network.server), [macOS App Sandbox](https://developer.apple.com/documentation/xcode/configuring-the-macos-app-sandbox)). TN3179 also states that macOS does not need the multicast entitlement.

The Viewer additionally creates a stable installation identifier, publishes its stable truncated `vid`, and sends the full identity in the NearWire hello path. The repository already classifies the equivalent installation correlation as linked Device ID for App functionality, but this Core-only Viewer target will not inherit the base SDK's privacy resource. No Viewer privacy-manifest decision is recorded. The foundation UI says the pairing code is not authentication, but does not positively tell the user that the code and stable `vid` are advertised to nearby Bonjour browsers.

**Required remediation:** make the build contract explicit. Prefer App Sandbox with only incoming-network server access, no network-client or multicast entitlement unless later behavior proves it necessary, no Keychain-sharing group, and a concrete signed development/test configuration so entitlement and local-network behavior can be exercised. Include a bounded English `NSLocalNetworkUsageDescription` and `_nearwire._tcp` in `NSBonjourServices`. Add a Viewer-owned `PrivacyInfo.xcprivacy` declaring the same linked Device ID/App-functionality/tracking-false decision as the repository's equivalent installation-ID behavior, with no tracking-domain or Required Reason entries unless final APIs require them; alternatively, record an authoritative contrary rationale before source work. Add a concise UI/operator statement that the pairing code and `vid` are nearby-discoverable identifiers, not secrets or Viewer authentication. Validate the built app's Info.plist, signed entitlements, privacy resource, and local-network denial/recovery path.

## Verified Boundaries and Proportionality

- The exact UI wording `TLS encrypted; Viewer identity is not authenticated` is accurate. It neither implies PKI/pinning nor downgrades the mandatory TLS 1.3, ALPN, TCP, and presented-leaf Basic X.509 checks. The pairing code is consistently excluded from password/credential semantics.
- Pairing generation is appropriately narrow: `SecRandomCopyBytes`, rejection sampling over the canonical alphabet, one memory-only code per generation, user-initiated clipboard only, and no log, analytics, Keychain, or session persistence.
- Bonjour publishes only the instance name and validated `vid`; exact-name registration and bounded conflict retry prevent a conflict-renamed service from being presented as usable. The finding above concerns positive disclosure and app metadata, not extra TXT leakage.
- Window ownership, exact generation tokens, stale-callback rejection, bounded shutdown, and no menu-bar/daemon lifetime appropriately constrain idle work. Keeping active sessions, persistence, Event flow, search, charts, and multi-device policy out of this foundation is the correct scope.
- A Viewer-only Apple-framework implementation, manual Xcode project, and placeholder handoff sink are proportionate. No backend service, general certificate authority, database, project generator, or active-session subsystem is needed for this change.
- Strict OpenSpec validation currently passes, but that validates artifact structure rather than the substantive bounds above.
